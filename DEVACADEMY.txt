# Developer Academy Course Creation Rules for Asset Tracker Template

## Course Context and Scope

You are building a Nordic Developer Academy course on the **Asset Tracker Template (ATT)**. This course should:

- Follow DevAcademy style, structure, and pedagogical approach
- Be complementary to the Cellular IoT Fundamentals course (prerequisite)
- Focus on practical, hands-on learning with real hardware (Thingy:91 X, nRF9151 DK)
- Cover ATT-specific topics: modular architecture, state machines (SMF), zbus messaging, customization

## Course Structure Requirements

### Lesson Organization
Each lesson must include:

1. **Introduction Section**
   - Clear learning objectives (3-5 bullet points)
   - Brief overview of the topic and its relevance
   - Prerequisites (SDK version, hardware, prior lessons)

2. **Theory Section**
   - Explain key concepts with clarity and simplicity
   - Use diagrams where appropriate (reference existing SVGs in docs/images/)
   - Link to official documentation for deeper exploration
   - Keep text concise but comprehensive

3. **Hands-On Exercises** (1-3 per lesson)
   - Provide clear step-by-step instructions
   - Include code skeletons where appropriate
   - Specify expected outcomes and how to verify
   - Offer troubleshooting tips for common issues

4. **Solution Discussion**
   - Explain the reasoning behind implementation choices
   - Discuss alternative approaches when relevant
   - Provide solution code or reference to solution files
   - Encourage self-assessment and comparison

### Exercise Format Guidelines

Each exercise should follow this pattern:

```
### Exercise X: [Clear, Action-Oriented Title]

**Objective:** [One sentence describing what the learner will accomplish]

**Duration:** [Estimated time: 15-30 minutes typical]

#### Task Description
[Clear description of what needs to be implemented]

#### Step-by-Step Instructions
1. [First step with specific file/function to modify]
2. [Second step with code context]
3. [Build and test instructions]
4. [Verification steps with expected output]

#### Code Hints
[Provide code skeletons or key function signatures when helpful]

#### Expected Outcome
[Describe what should happen when successful]

#### Troubleshooting
[Common issues and solutions]

#### Solution Discussion
[Explain the implementation, design choices, and alternatives]
```

## Content Guidelines

### Writing Style
- **Simplicity:** Use clear, direct language. Avoid unnecessary jargon.
- **Brevity:** Be concise but complete. Every sentence should add value.
- **Practicality:** Focus on what developers need to know to use ATT effectively.
- **Encouragement:** Use positive, supportive language. Celebrate progress.

### Code Examples
- Always test code examples for correctness
- Use proper formatting and indentation
- Include necessary includes and context
- Comment complex logic appropriately
- Reference actual ATT source code when possible

### Technical Accuracy
- Verify all technical details against the ATT codebase
- Use correct Kconfig option names and values
- Reference correct file paths and function names
- Test build commands and instructions

## Implementation Instructions

### File Organization
Create this structure in devacademy/

```
devacademy/
├── README.md (course overview and navigation)
├── lesson-1/
│   ├── lesson.md (lesson content)
│   ├── exercises/
│   │   ├── exercise-1.md
│   │   └── exercise-2.md
│   └── solutions/
│       ├── solution-1/
│       └── solution-2/
├── lesson-2/
│   └── ...
├── assets/
│   ├── images/
│   └── diagrams/
└── resources/
    └── reference-links.md
```

### Resource Utilization
- **Use nordic-db-mcp extensively** for DevAcademy style verification
- **Reference existing ATT documentation** in docs/ folder
- **Link to official NCS documentation** for deeper dives
- **Reuse diagrams** from docs/images/ where applicable
- **Reference actual source code** from app/src/modules/

### Quality Assurance
Before considering a lesson complete:
1. Verify all technical details against the codebase
2. Test all exercises and solutions
3. Check for clarity and simplicity
4. Ensure consistent formatting
5. Validate all links and references
6. Review against DevAcademy standards via nordic-db-mcp

## Key Success Criteria

✓ **User-Friendly:** Clear navigation, progressive difficulty, supportive tone
✓ **Rewarding:** Visible progress, working examples, "aha moments"
✓ **Practical:** Real-world applications, tested on hardware
✓ **Self-Paced:** Complete explanations, no assumed knowledge beyond prerequisites
✓ **Professional:** Consistent style, accurate content, proper formatting

## Iteration Process

1. **Research:** Use nordic-db-mcp to understand DevAcademy patterns
2. **Plan:** Create lesson outline and exercise structure
3. **Draft:** Write lesson content with exercises
4. **Implement:** Create solution code and test
5. **Verify:** Check against ATT codebase and DevAcademy style
6. **Refine:** Simplify, clarify, and polish
7. **Validate:** Final review against quality criteria

## Important Reminders

- **Simplicity over completeness:** Better to explain one thing well than many things poorly
- **Show, don't tell:** Use code examples and practical exercises
- **Test everything:** Every exercise must work on real hardware
- **Reference the source:** Link to ATT docs and code frequently
- **Stay current:** Use NCS v3.1.0+ and latest ATT features
- **Be encouraging:** Learning embedded systems is challenging; support the learner



The agent should keep iterating until completion of the task.
